# TIPO: Software y SaaS

## 1. Descripcion

Servicios de desarrollo de software a medida, licenciamiento de software, suscripciones SaaS, implementacion de sistemas y soporte tecnico continuo.

## 2. Riesgo fiscal inherente

⬜ Bajo | ☑️ **Medio** | ⬜ Alto | ⬜ Muy alto

### Ejemplos de operaciones tipicas:
- Desarrollo de aplicacion web a medida
- Implementacion de ERP/CRM
- Suscripcion anual a software SaaS
- Integracion de sistemas
- Soporte y mantenimiento mensual
- Licencias de software

## 3. Checklists por pilar

### 3.1 Razon de negocios (A1)

Preguntas obligatorias en SIB:
- [ ] ¿Que proceso de negocio automatiza o mejora?
- [ ] ¿Cuanto tiempo/costo ahorra vs proceso manual?
- [ ] ¿Cuantos usuarios lo utilizaran?
- [ ] ¿Por que este software y no alternativas?

### 3.2 Beneficio economico (A1/A5)

Ejemplos de BEE aceptables:
- "El sistema reduce tiempo de facturacion de 4 horas a 30 minutos diarios = 175 horas/mes ahorradas"
- "El CRM incremento seguimiento de leads en 40%, aumentando conversion de 5% a 8%"
- "La automatizacion elimino 2 FTE de tareas manuales = $600K anuales de ahorro"

### 3.3 Materialidad (A3/A7)

Evidencias minimas:
- [ ] Contrato o SOW detallado
- [ ] Repositorio de codigo (si es desarrollo)
- [ ] Tickets/historias de usuario
- [ ] Pruebas de aceptacion (UAT)
- [ ] Acta de go-live
- [ ] Capturas de pantalla del sistema funcionando
- [ ] Logs de uso del sistema

### 3.4 Trazabilidad (A2/A7)

Documentos con sello de tiempo obligatorio:
- [ ] SOW firmado
- [ ] Commits/versiones de codigo
- [ ] Actas de sprint review
- [ ] Documento de go-live
- [ ] Facturas mensuales (si es SaaS)

## 4. Red flags tipicos

- Software que nadie usa despues de implementado
- Licencias para mas usuarios de los que existen
- Desarrollo sin repositorio de codigo
- Sin pruebas de aceptacion documentadas
- Cobros excesivos de "customizacion"
- Proveedor sin equipo tecnico verificable

## 5. Notas para defensa (A7)

### Argumentos estandar que funcionan:
- "El sistema esta en produccion desde [fecha], con [X] usuarios activos"
- "Se adjuntan logs de uso que demuestran utilizacion diaria"
- "El codigo fuente esta en repositorio con historial de commits"
- "Las pruebas UAT fueron firmadas por el area usuaria"

### Errores recurrentes a evitar:
- No guardar el repositorio de codigo
- No documentar pruebas de aceptacion
- Pagar por fases no entregadas
- No conservar evidencia de uso posterior
